# Практична робота №28. Створення формальної специфікації програми
### Розділ VІІ. Методи доведення та верифікації програмних систем
### Тема 16. Доведення правильності програм. Формальні специфікації
### Мета: Описати функціональність обраного модуля через preconditions, postconditions та invariants.
## Завдання
1. Обрати один use-case вашого проєкту.
2. Сформулювати формальний контракт операції.
3. Описати інваріанти стану.
4. Підготувати приклади допустимих/недопустимих сценаріїв.
## Хід роботи
### Крок 1. Визначення меж операції
На цьому етапі фіксуємо дію, щоб специфікація була перевірюваною.
Обраний use-case: Бронювання одного місця на подію (Event) зареєстрованим користувачем.
Вхідні дані: user_id (ідентифікатор користувача), event_id (ідентифікатор події).  
Вихід: Створений об'єкт бронювання (Booking) у разі успіху, або генерація помилки. 
Побічні ефекти: Зменшення кількості доступних місць на подію в базі даних, створення нового запису в таблиці бронювань.
## Крок 2. Оформлення контракту
Тут ми описуємо базу для подальшого доведення коректності операції.

### Operation: book_seat(user_id, event_id)
### Preconditions:
Event.exists(event_id) == true (Подія існує в системі).

User.exists(user_id) == true (Користувач зареєстрований).

Event.get(event_id).available_seats > 0 (На подію є хоча б одне вільне місце).

Booking.exists(user_id, event_id) == false (Користувач ще не бронював місце на цю подію - запобігаємо випадковим дублям квитків).

Postconditions:
Event.get(event_id).available_seats == Event.get(event_id).available_seats@pre - 1 (Кількість вільних місць для цієї події зменшилась рівно на 1).

Booking.exists(user_id, event_id) == true (У системі успішно створено запис про бронювання).

State invariants:

∀ event ∈ Events: event.available_seats >= 0 (Кількість вільних місць фізично не може бути від'ємною - "овербукінг" на рівні бд заборонений).

Event.get(event_id).available_seats + Booking.count(event_id) == Event.get(event_id).total_capacity (Жорсткий закон збереження: сума вільних місць та всіх проданих квитків завжди дорівнює загальній місткості залу).
## Крок 3. Перевірні сценарії
Складаємо 3 позитивні й 3 негативні кейси, щоб специфікація була операційно перевірною.

### Позитивні сценарії (Допустимі):
1. Стандартне бронювання: Користувач бронює квиток на концерт, де є 100 вільних місць. Результат: Створюється бронь, кількість вільних місць стає 99.
2. Бронювання останнього місця: На подію залишилося рівно 1 місце (available_seats == 1). Результат: Операція успішна, кількість вільних місць стає 0, інваріант не порушено.
3. Бронювання іншої події: Користувач уже має квиток на подію А, але тепер викликає операцію для події Б. Результат: Бронювання проходить успішно, оскільки передумова про відсутність дублів перевіряється в розрізі конкретної події.
### Негативні сценарії (Недопустимі):
1. Sold Out (Немає місць): Користувач намагається забронювати квиток на подію, де available_seats == 0. Результат: Порушення передумови available_seats > 0. Система відхиляє запит.

2. Повторне бронювання: Користувач випадково клікає кнопку двічі або свідомо намагається купити другий квиток на той самий event_id з цього ж акаунту. Результат: Порушення передумови Booking.exists(...) == false. Транзакція скасовується.

3. Неіснуюча подія: Зловмисник або баг у фронтенді передає event_id, якого немає в базі (наприклад, подію вже видалили). Результат: Порушення передумови Event.exists(event_id) == true. Операція переривається помилкою "Not Found".
